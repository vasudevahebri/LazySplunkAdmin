Splunk Enterprise Deployment & Daily Ops Studio

User Guide — PS Config Factory Edition

Aligned to file: Splunk_Enterprise_Daily_Ops_Studio_PS_Config_Factory.html
Updated: 2026-04-13

1. Purpose

This guide explains how to use the updated Splunk Enterprise Deployment & Daily Ops Studio application as a Professional Services style engineering workbook.

The app is designed to help Splunk engineers do four things in one place:

  1. capture customer requirements and component inventory,
  2. build and validate a real Splunk topology,
  3. decide where each app, TA, and configuration should live,
  4. generate PS-style config outputs and placement guidance for delivery teams.

The updated version adds a stronger PS Config Factory workflow so the tool can convert generic org_* naming into customer-specific folder names, produce engineer-readable conf snippets, and tell the engineer exactly where each app or file should be staged.

2. What changed in this version

The updated app now includes the following PS-oriented improvements:

  • A new Customer short name field in Customer profile and platform settings.
  • Automatic replacement of generic org_ prefixes with customer-specific folder names.
  • A dedicated PS config factory and placement map section.
  • A generated bundle manifest that maps original folders to customer-prefixed PS folder names.
  • A generated conf pack for common deployment, forwarding, licensing, cluster, and security baselines.
  • Placement guidance that separates:
    • deployment-server apps,
    • cluster manager peer bundles,
    • search head cluster deployer apps,
    • manager-local apps,
    • peer-local overlays.
  • A more explicit PS workflow that connects:
    • profile inputs,
    • component inventory,
    • data source onboarding,
    • app and TA placement,
    • playbook execution,
    • validation and troubleshooting.

3. Recommended engineer workflow

Use the app in this sequence.

Step 1 — Start with the customer profile

Complete the Customer profile and platform settings section first.

At minimum, populate these fields:

  • Deployment name
  • Customer name
  • Customer short name
  • Environment
  • Splunk version
  • Topology pattern
  • Platform
  • Enterprise home
  • Forwarder home
  • Receiver port
  • Deployment poll interval
  • TLS and clustering settings

Step 2 — Define sites and component roles

Use the Site planner and customer-request builder and Component inventory sections to define the actual estate.

Typical roles include:

  • Load balancer
  • Cluster manager
  • Deployer
  • Deployment server
  • Monitoring console
  • License manager
  • Search head
  • Indexer
  • Heavy forwarder
  • Universal forwarder

These roles are used later by the app to generate stage paths, manager URIs, deployer targets, and placement decisions.

Use Data source inventory and onboarding inputs to capture onboarding intent.

This is where the app decides:

  • collection tier,
  • parsing tier,
  • search tier,
  • likely TA split,
  • ownership,
  • validation steps,
  • rollback notes.

Step 4 — Review architecture and placement

Use these sections before generating configs:

  • Topology summary and architecture warnings
  • Visual HLD
  • Visual LLD and connection matrix
  • App and add-on placement advisor

These sections tell you whether the topology is consistent and whether an app belongs on the deployment server, deployer, manager-apps, peer-local, or local search tier.

Step 5 — Generate PS naming and conf outputs

Use PS config factory and placement map after profile, sites, and components are complete.

This section gives you:

  • customer-prefixed app names,
  • bundle manifest,

Step 6 — Execute with the TA matrix and playbook

After the config factory, continue with:

  • TA and app delivery matrix

Use the final sections during rollout and handoff:

  • Shift Handover Notes

4. Customer short name and PS naming standard

The Customer short name field is now a primary input.

The app uses it to replace generic PS starter names with customer-specific folder names.

Example naming conversions

Original name Generated PS-style name
TA-org_splunk TA-<customerShortName>_splunk
org_all_forwarder_outputs <customerShortName>_all_forwarder_outputs
org_all_deploymentclient <customerShortName>_all_deploymentclient
org_cluster_indexer_base <customerShortName>_cluster_indexer_base
org_multisite_manager_base <customerShortName>_multisite_manager_base
org_site_n_indexer_base <customerShortName>_<site>_indexer_base

Naming rules used by the app

The generator sanitizes the short name so it remains safe for Splunk app-folder usage:

  • spaces are removed or normalized,
  • non-alphanumeric separators are converted to underscores,
  • leading or trailing separators are removed,
  • the result is kept predictable for packaging and deployment.

Recommended naming approach

Use a short, stable customer code such as:

  • acme
  • cba
  • finops
  • globex

Avoid long descriptive names, spaces, and punctuation-heavy labels.

6. How to use the PS Config Factory

The PS config factory and placement map section should be used after the core environment details are in place.

6.1 Inputs required

For the best output, define at least:

  • Customer short name
  • Enterprise home
  • Receiver port
  • Deployment server
  • Cluster manager
  • License manager
  • Deployer
  • Search heads
  • Indexers
  • Sites and site names
  • Cluster and SHC settings

If a required component is missing, the app still generates output, but some values remain placeholders.

6.2 What the section shows

The section contains three practical parts.

A. PS naming convention panel

Shows how generic starter names are converted to customer-prefixed names.

B. Component inputs used by the generator

Shows which real component records are being used to build stage paths and target values.

C. Bundle safety rules

Highlights rules such as:

  • manager-local apps stay local,
  • peer-local overlays stay local to the peer,
  • SHC apps go through the deployer,
  • DS apps go through deployment-apps.

D. Renamed bundle manifest

The manifest maps:

  • source pack,
  • original folder,
  • generated PS folder,
  • placement target,
  • stage path,
  • delivery notes.

E. Generated config cards

Each config card includes:

  • config title,
  • target app name,
  • target file,
  • stage path,
  • placement note,
  • copyable conf content,
  • a download button for the individual file.

F. Download buttons

The section currently supports:

  • Download PS manifest
  • Download generated conf pack
  • Download file for each generated config

The current app generates guidance for common Splunk PS baselines, including:

  • Deployment client baseline
  • Fixed receiver outputs baseline
  • Intermediate forwarder internal-index overlay
  • Deployment server index-and-forward overlay
  • License peer baseline
  • Deployment client TLS trust overlay
  • Deployment server secure-server baseline

6.4 How to use the generated content

Use the generated configs as engineer starting points, not as blind production drops.

Before deployment, confirm:

  • real hostnames,
  • real secrets,
  • TLS certificate locations,
  • site names,
  • retention and volume paths,
  • correct serverclass targeting,
  • correct SHC and cluster membership.

7. Placement standards used by the app

The updated app now follows a PS-style split between release channels.

Content type Standard placement used by the app Notes
Deployment-managed client apps $SPLUNK_HOME/etc/deployment-apps/<app> on the deployment server Used for UFs and selected Enterprise clients
Search head cluster apps $SPLUNK_HOME/etc/shcluster/apps/<app> on the deployer Use the deployer, not the deployment server
Common indexer-cluster peer bundles $SPLUNK_HOME/etc/manager-apps/<app> on the cluster manager Validate and apply as a cluster bundle
Manager-local apps $SPLUNK_HOME/etc/apps/<app> on the cluster manager Do not place manager-local content in manager-apps
Peer-local overlays $SPLUNK_HOME/etc/apps/<app> on the peer Site-local overlays should remain local
Standalone or local search-tier apps $SPLUNK_HOME/etc/apps/<app> on the search tier Use when SHC is not enabled

Practical rule

Treat the following as separate release channels:

  1. Deployment server
  2. Cluster manager
  3. SHC deployer
  4. Local-only app placement

Do not mix these channels just because a folder name looks similar.

8. TA and app packaging guidance

The app assumes a PS-style split of add-ons when needed.

Collection package

Typical contents:

  • inputs.conf
  • outputs.conf
  • deploymentclient.conf
  • collector-side scripts or modular input settings

Parsing package

Typical contents:

  • props.conf
  • transforms.conf
  • indexes.conf
  • tier-specific parsing rules

Search package

Typical contents:

  • search-time props.conf
  • search-time transforms.conf
  • tags
  • eventtypes
  • macros
  • lookups
  • saved searches
  • data models
  • dashboards

Important sanitation rule

Do not keep collection files on the search tier unless that tier is intentionally collecting data.

9. Engineer operating procedure

For each customer implementation, use this working pattern:

  1. Load or define the customer profile.
  2. Set the customer short name.
  3. Define sites.
  4. Enter all components.
  5. Enter data sources.
  6. Review topology warnings.
  7. Review HLD and LLD.
  8. Review app and TA placement.
  9. Open PS Config Factory.
  10. Download the manifest.
  11. Download the generated conf pack.
  12. Review the TA delivery matrix.
  13. Execute the deployment playbook.
  14. Run validation packs.
  15. Capture handover notes.

This ensures the naming, placement, and rollout sequence remain aligned.

10. Common mistakes to avoid

Avoid these common engineering mistakes:

  • Leaving org_ names unchanged after moving to customer delivery.
  • Using the deployment server for SHC member app deployment.
  • Putting cluster-manager local settings in manager-apps.
  • Pushing peer-local site overlays as shared cluster bundles.
  • Keeping search-only knowledge on the parsing or collection tier without a reason.
  • Forgetting to update hostnames, secrets, CA paths, and storage paths.
  • Treating generated files as final production-ready content without environment review.
  • Applying TA content to every tier instead of splitting collection, parsing, and search responsibilities.
