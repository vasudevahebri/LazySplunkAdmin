1. What this SOP is for

Use this SOP when the tool has already told you what the target app or file should be and you now need to execute the change like a Splunk PS engineer: safely, in the correct repository, with the correct bundle action, validation, and rollback discipline.

2. Golden decision table

Situation Correct place Correct control plane Typical files Do not do
UF or HF client app etc/deployment-apps/<app> Deployment Server inputs.conf, outputs.conf, deploymentclient.conf, client trust files Do not use this path for SHC member apps
Clustered peer bundle etc/manager-apps/<app> Cluster Manager server.conf, indexes.conf, peer-side parsing files Do not leave shared peer settings as ad hoc local edits
SHC member app etc/shcluster/apps/<app> Deployer search-time knowledge, lookups, dashboards, macros, search-tier configs Do not push SHC apps with Deployment Server
Manager-local or peer-local app etc/apps/<app> on the target node Local change on the right host local-only overlays, manager local, peer site overlay Do not place manager-local settings in manager-apps

3. Pre-change SOP

Step A — Freeze the target and owner

  1. Confirm the exact business request.
  2. Confirm the exact target tier: deployment client, indexer cluster, SHC, or local app.
  3. Confirm the change owner and rollback owner.
  4. Capture the current expected state from the tool:
    • target app name,
    • stage path,
    • generated conf file,
    • validation query or CLI,
    • affected hosts.

Step B — Confirm the control-plane safety rules

  • The Deployment Server must not be its own deployment client.
  • SHC member content must be pushed from the deployer, not the Deployment Server.
  • Shared peer changes must go through manager-apps and the cluster bundle method.
  • Search-tier packages should not accidentally carry collection stanzas like inputs.conf unless the search tier truly collects local data.
  • Peer-local site overlays stay local on the peer and are not part of a shared common-peer bundle.

Step C — Take a quick baseline

Use the relevant command builder or PS CLI section and capture:

splunk btool check
splunk show cluster-status --verbose
splunk show shcluster-status --verbose
splunk list deploy-clients

Keep the before-state in the change record or handover notes.

4. Rollout SOP by change type

A. Deployment Server rollout

  1. Stage the app under etc/deployment-apps/<app>.
  2. Place the generated file into the correct local/ path.
  3. Review targeting in serverclass.conf or the forwarder-management mapping.
  4. Reload the Deployment Server.
  5. Validate the effective config on one real client with btool.
  6. Confirm that the source still forwards and the client still phones home.

B. Cluster Manager rollout

  1. Stage the app under etc/manager-apps/<app>.
  2. Place the generated file under the correct bundle app path.
  3. Run validate cluster-bundle --check-restart.
  4. Apply the cluster bundle.
  5. Confirm peer-side effective config on at least one indexer with btool.
  6. Confirm cluster health after the change.

C. SHC deployer rollout

  1. Stage the app under etc/shcluster/apps/<app> on the deployer.
  2. Ensure the package is search-tier safe.
  3. Run SHC health check before the push.
  4. Apply the SHC bundle from the deployer.
  5. Validate effective config on one member.
  6. Re-check SHC health and app presence on members.

D. Local app rollout

  1. Copy the file into etc/apps/<app>/local on the intended host or hosts.
  2. Run btool on the same host.
  3. Restart only the affected host or tier.
  4. Confirm the precise behavior that changed.

5. Post-change validation SOP

Validate in four layers:

Layer 1 — Control plane

  • Deployment clients visible and not stale.
  • Cluster bundle status clean.
  • SHC status clean.
  • No new severe errors in splunkd.log.

Layer 2 — Effective config

  • btool shows the expected setting on the correct tier.
  • No unexpected override from another app.
  • The app name and path match the PS naming convention.

Layer 3 — Data or service result

  • Data reaches the intended index.
  • Search-tier knowledge behaves as expected.
  • Listener, HEC, API, or scripted input still works.
  • No regression on adjacent sources.

Layer 4 — Operational impact

  • Queues remain healthy.
  • License usage is not spiking unexpectedly.
  • Scheduler health is normal after the change.
  • User-visible dashboards and alerts still function.

6. Rollback SOP

Rollback from the same path you used for deployment.

  • If you changed deployment-apps, back out from deployment-apps and reload the Deployment Server.
  • If you changed manager-apps, remove or revert the bundle content, validate again, and re-apply the cluster bundle.
  • If you changed shcluster/apps, revert on the deployer and push the bundle again.
  • If you changed a local app, restore the previous file on the local host and restart the local instance.

Always re-run the same validation commands after rollback. A rollback that is not validated is incomplete.

7. Real-time troubleshooting SOP

Symptom 1 — Config landed but behavior did not change

  1. Run btool on the real winning tier.
  2. Check for a higher-precedence override.
  3. Confirm the app was pushed through the correct control plane.
  4. Compare with the last known good app version.

Symptom 2 — App reached the wrong tier

  1. Check whether the package should have been DS, cluster manager, deployer, or local.
  2. Remove the app from the wrong repository.
  3. Re-stage it in the correct repository.
  4. Re-run the proper reload or bundle action.

Symptom 3 — Data stopped after a rollout

  1. Check listener or forwarding health.
  2. Check outputs and deployment-client settings with btool.
  3. Check parsing tier props.conf and transforms.conf if the issue is index-time.
  4. Check indexer or collector logs for queue, TLS, or routing failures.

Symptom 4 — Cluster or SHC health changed after a push

  1. Check bundle status.
  2. Check merged config on one healthy and one unhealthy node.
  3. Compare the pushed app content against the intended package scope.
  4. Back out the last bundle if the drift began with the latest push.

8. Handover checklist

  • Change summary in one sentence.
  • App or file changed.
  • Exact repository used.
  • Exact hosts affected.
  • Commands run.
  • Validation result.
  • Known watch items.
  • Rollback state and owner.

9. No-go list

  • Do not use the Deployment Server for SHC member apps.
  • Do not push manager-local settings in manager-apps.
  • Do not keep search-only knowledge and collection stanzas in the same SHC package unless the use case is intentional.
  • Do not assume file copy equals active config; always prove with btool.
  • Do not close the change until post-change validation and handover are complete.
